What Does a PM Do?
The glue between business, technology, and users
🎭 The PM Role
A Product Manager defines what to build and why — not how. They sit at the intersection of three worlds:

• Business — ensuring the product generates value and meets company goals
• Technology — understanding what's feasible and working with engineers
• Users — deeply understanding customer needs and pain points

The PM doesn't manage people — they manage the product vision and roadmap.
🌍 Real World Example
Product Management at Spotify
Spotify's PM for the "Discover Weekly" playlist feature had to: understand user frustration with music discovery (users), work with data scientists and engineers to build a recommendation model (technology), and convince leadership it would increase engagement and retention (business). They defined the "what" — a personalised weekly playlist — while the engineering team built the "how".
📋 PM Responsibilities
• Define product vision & strategy
• Build and maintain the roadmap
• Gather & prioritise requirements
• Write user stories and acceptance criteria
• Align stakeholders
• Measure product success (KPIs)
• Make build vs. buy vs. partner decisions
🛠️ PM Skills
• Communication — translate between tech & business
• Empathy — deeply understand users
• Data literacy — make evidence-based decisions
• Prioritisation — say no gracefully
• Strategic thinking — see the big picture
• Facilitation — run effective meetings

🏃 Agile Fundamentals
Agile is a mindset, not a process. It's how modern software teams deliver value faster, adapt to change, and build the right thing — not just build the thing right.
⏱ ~35 min
🎯 3 Quizzes
⭐ 180 XP

The Agile Manifesto
4 values and 12 principles (2001)
The 4 Agile Values
Individuals & interactions
over processes and tools
Working software
over comprehensive documentation
Customer collaboration
over contract negotiation
Responding to change
over following a plan
💡
The right side items still have value — Agile just says the left side items matter more. It's about balance, not abandonment.
🌍 Real World Example
Waterfall vs Agile at Nokia vs Apple
Nokia used Waterfall in 2007: 2-year phone development cycles with full specs locked upfront. By the time phones shipped, the market had moved. Apple used iterative development — the first iPhone shipped with limited features but the team responded to user feedback rapidly with iOS updates. Within 3 years, the App Store had transformed the product based on what customers actually wanted, not what was planned 2 years before.

📖 User Stories & INVEST
User stories are the currency of Agile. They capture requirements from the user's perspective — not the system's — and keep the team focused on delivering real value.
⏱ ~35 min
🎯 3 Quizzes
⭐ 180 XP
✍️ Interactive
📝
The User Story Format
Three-part sentence that keeps focus on user value
User Story Template
As a [type of user],
I want to [perform some action],
so that [I achieve some goal/benefit].
💡
The "so that" clause is critical — it states the why. Without it, teams build features without understanding the purpose, leading to solutions that technically work but miss the point.
✍️ Build Your Own User Story
As a
I want to
So that
✅
INVEST Criteria
What makes a good user story?
I
Independent
Can be developed and delivered without depending on other stories
N
Negotiable
Details can be discussed and adjusted — not a fixed contract
V
Valuable
Delivers clear value to users or the business — not just technical work
E
Estimable
Team can estimate the effort — if not, it needs more splitting or clarification
S
Small
Completable within one sprint. Epic → Stories → Tasks
T
Testable
Clear acceptance criteria — you know when it's done
🌍 Real World Example
Acceptance Criteria at Monzo Bank
User story: "As a Monzo user, I want instant spending notifications so that I can track my money in real-time."

Acceptance criteria (Given/When/Then):
• Given I have the Monzo app installed, When a transaction occurs, Then I receive a push notification within 3 seconds
• Given my phone is on silent, When a transaction occurs, Then the notification still appears in the notification centre
• Given the transaction fails, When it's declined, Then I receive a notification explaining why

🗓️ Agile Rituals & Estimation
Agile ceremonies are the structured touchpoints that keep teams aligned, unblocked, and continuously improving. Each ritual has a clear purpose — done badly, they're time-wasters; done well, they're multipliers.
⏱ ~30 min
🎯 2 Quizzes
⭐ 150 XP
🎭
The 4 Scrum Ceremonies
📋 Sprint Planning
When: Start of each sprint
Duration: 2–4 hours (for 2-week sprint)
Purpose: Select backlog items for the sprint, define the Sprint Goal, break stories into tasks

Output: Sprint Backlog — the team's commitment
☀️ Daily Standup
When: Every day, same time
Duration: 15 minutes max (standing!)
3 questions:
• What did I do yesterday?
• What will I do today?
• Any blockers?

Purpose: Synchronise, not report to manager
🎯 Sprint Review
When: End of sprint
Duration: 1–2 hours
Purpose: Demo working software to stakeholders, gather feedback

Key rule: Only show working software — no slidedecks pretending things are done
🔄 Retrospective
When: After Sprint Review
Duration: 1–1.5 hours
3 questions:
• What went well? ✅
• What could improve? 🔧
• Action items? 📌

The engine of continuous improvement
🃏
Story Point Estimation
Estimating effort, not time
🃏 Planning Poker
Teams estimate user stories in story points — a relative measure of effort, complexity, and uncertainty — using Fibonacci numbers: 1, 2, 3, 5, 8, 13, 21.

Process: Everyone secretly picks a card → reveal simultaneously → discuss outliers → re-estimate → repeat until consensus.

Why Fibonacci? The gaps between numbers grow with uncertainty. A 13-point story isn't exactly 13x harder than a 1 — it's "much harder and much less certain."
🃏 Estimate These Stories — What Would You Pick?
Story: "As a user, I want to log in with email and password."
Story: "As an analyst, I want an AI-powered anomaly detection dashboard."

🎯 Prioritisation & MVP
A PM's most important skill is saying no. Learn MoSCoW, the MVP concept, and how to make tough trade-off decisions that keep teams focused on what truly matters.
⏱ ~30 min
🎯 2 Quizzes
⭐ 150 XP
🎮 Interactive
🏆
MoSCoW Prioritisation
A framework for making trade-offs explicit

🔴 Must Have

Non-negotiable. Without these, the product fails.
User authentication
Payment processing
Core search feature

🟡 Should Have

Important but not critical for launch. Include if time allows.
Email notifications
Export to PDF
Advanced filters

🔵 Could Have

Nice to have. Low impact if left out.
Dark mode
Keyboard shortcuts
Animated transitions

⚪ Won't Have

Explicitly out of scope. Sets expectations.
Multi-language support
Offline mode
AR features
💡
Golden Rule: Must Haves should be ≤60% of the total effort. If everything is a Must Have, nothing is — prioritisation has failed.
🌱
Minimum Viable Product (MVP)
The smallest thing you can build to test your biggest assumption
🌱 What is an MVP?
An MVP is the minimum set of features needed to deliver value to early customers and generate validated learning about the product.

The goal is NOT to build a half-baked product — it's to test your riskiest assumption as cheaply as possible before committing to full development.

🌍 Real World Example
Airbnb's MVP — A Website and a Camera
Airbnb's founders needed to test: "Will strangers rent out their homes to other strangers?" Their MVP was a basic website plus going door-to-door in New York with a camera to photograph hosts' spaces. No complex matching algorithm, no payment system, no trust features. They just needed to validate the core assumption. Once confirmed, they built the full platform. This saved years of building the wrong thing.

💡 Design Thinking
A human-centred approach to innovation. Design Thinking gives teams a structured process for deeply understanding user problems and generating creative solutions — before writing a single line of code.
⏱ ~30 min
🎯 2 Quizzes
⭐ 150 XP
💎
The 5-Stage Process
Non-linear — you can loop back at any stage
🫀
Empathise
Observe, interview, deeply understand users' real problems
🎯
Define
Synthesise insights into a clear problem statement (HMW)
💭
Ideate
Generate many ideas — quantity over quality. No filtering yet!
🏗️
Prototype
Build rough, cheap versions — sketches, cardboard, Figma mockups
🔬
Test
Get feedback from real users on the prototype — then iterate
📝 "How Might We" Statements
The bridge between Empathise and Ideate. Frame the problem as a question starting with "How might we..." — it opens up possibilities without prescribing a solution.

❌ Problem: "Users can't find products easily"
❌ Solution frame: "We need a better search bar"
✅ HMW: "How might we help users discover the right product without knowing what to search for?"

The HMW reframe opens up solutions like: visual browsing, AI recommendations, quiz-based matching, curated collections — not just "a better search bar."
🌍 Real World Example
IDEO and the NHS Waiting Experience
IDEO applied Design Thinking to NHS waiting rooms. Empathise: They shadowed patients and observed anxiety, confusion about wait times, and isolation. Define: "How might we make waiting feel less uncertain and more human?" Ideate: 50+ ideas generated. Prototype: Simple digital boards showing estimated wait times + a small café area. Test: Patient satisfaction scores rose 34%. The insight — patients didn't mind waiting, they hated not knowing how long — only emerged through deep empathy, not assumption.

🗺️ User Journey Mapping
A user journey map visualises every step a customer takes to achieve a goal — their actions, thoughts, emotions, and pain points. It builds shared empathy across the entire team.
⏱ ~25 min
🎯 2 Quizzes
⭐ 120 XP
🗺️
Anatomy of a Journey Map
Stage 🔍 Awareness 🤔 Consideration 💳 Purchase 🎓 Onboarding 🔁 Retention
User Actions Sees ad on LinkedIn Reads reviews, compares tools Signs up for free trial Sets up account, watches tutorial Uses daily, invites colleagues
Thoughts "Could this help my team?" "Is it worth the price?" "I hope setup is easy" "Where do I start?" "I can't work without this now"
Emotion 😐 🤔 😬 😕 😊
Pain Points Ad is vague Pricing not clear Too many form fields Overwhelmed! No clear first step —
Opportunities Clearer value prop Transparent pricing page Reduce form to 3 fields Add guided setup wizard Referral programme
💡
The insight: The biggest drop-off isn't at signup — it's at onboarding. The user journey map makes this visible. Without it, the team might focus on improving the ad (awareness) while the real problem is the first-use experience.
🌍 Real World Example
Duolingo's Onboarding Redesign
Duolingo mapped their user journey and found: 70% of users who downloaded the app never completed a lesson. Journey map revealed the culprit — a lengthy registration form before users could try anything. They redesigned: let users try a lesson immediately, then register. Completion rate jumped from 20% to 65%. The fix cost 2 sprints. Without the journey map, they'd have kept optimising the wrong stage.

Lesson 8 of 9
🤖 Product Management for AI
AI products are fundamentally different from traditional software. They have probabilistic outputs, data dependencies, model drift, and ethical implications that require a new PM playbook.
⏱ ~35 min
🎯 3 Quizzes
⭐ 180 XP
⚡
What Makes AI Products Different
🧮 Traditional Software
• Deterministic: same input → same output
• "Done" when built
• Bugs are discrete and fixable
• Requirements can be fully specified
• Testing is straightforward
• Performance is stable over time
🤖 AI / ML Products
• Probabilistic: outputs have confidence, not certainty
• Never "done" — models drift as data changes
• Failures are gradual and statistical
• Requirements include data quality & volume
• Testing requires evaluation datasets
• Performance degrades without retraining
🌍 Real World Example
Amazon's Hiring Algorithm Failure
Amazon built an AI CV screener in 2014 and quietly discontinued it in 2018 — the model had learned to penalise CVs containing the word "women's" (e.g. "women's chess club") because it was trained on historical male-dominated hiring data. A PM with AI literacy would have demanded: bias testing on protected characteristics, model cards documenting limitations, and human-in-the-loop review for edge cases. Without PM oversight, the model shipped with systematic discrimination baked in.
The AI Product PM's Unique Responsibilities
📊
Data Strategy — What data do we need? How much? Where does it come from? How is it labelled? Is it representative?
🎯
Success Metrics — Define what "good" looks like before training. Accuracy? Precision? Recall? F1? Business KPIs vs. model KPIs.
⚖️
Responsible AI — Fairness, explainability, privacy (GDPR), bias testing. Especially critical in healthcare, finance, HR.
🔄
Model Lifecycle — Plan for monitoring, retraining, versioning. A model isn't a feature — it's a living system.
🤝
Human-in-the-Loop — Where should humans review AI decisions? High-stakes decisions (medical, legal, financial) almost always need human oversight.

🧪 Testing Principles
Testing isn't a phase at the end — it's built into every stage of development. Understanding testing types helps PMs write better acceptance criteria and have more productive conversations with QA engineers.
⏱ ~30 min
🎯 2 Quizzes
⭐ 150 XP
🔺
The Testing Pyramid
More tests at the bottom, fewer (but broader) at the top
🧑 Manual / Exploratory Human testers explore edge cases
🖥️ End-to-End Tests Simulate real user workflows — slow, expensive, valuable
🔗 Integration Tests Test that components work together correctly
⚡ Unit Tests Fast, isolated tests of individual functions — the foundation
⚡ Unit Tests
What: Test a single function in isolation
Speed: Milliseconds
Written by: Developers
Example: Test that calculate_discount(100, 0.2) returns 80.0
🔗 Integration Tests
What: Test how components work together
Speed: Seconds
Example: Test that the payment service correctly talks to the order database when a purchase is made
👤 User Acceptance Tests
What: Does it meet the user's requirements?
Who: PMs, stakeholders, real users
Example: "Given I add 3 items to cart, when I checkout, then my order confirmation email arrives within 1 minute"
📋
Acceptance Criteria & Definition of Done
✅ Definition of Done (DoD)
A shared team agreement on what "done" means for any user story. Prevents "I thought it was done" surprises.

Example DoD checklist:
☑ Code written and reviewed by a peer
☑ Unit tests written and passing
☑ Acceptance criteria verified by PM
☑ No new bugs introduced (regression tests pass)
☑ Deployed to staging environment
☑ Product Owner has signed off
☑ Documentation updated if needed
🌍 Real World Example
NHS COVID Test Booking System Failure
In 2020, the NHS Test and Trace booking system crashed on launch day — 1 million users hitting a system tested for 10,000. The system had passed all unit and integration tests but was never end-to-end load tested at production scale. A PM-led Definition of Done that included "load tested to 5× expected peak traffic" would have caught this before launch. Proper testing isn't just about bugs — it's about realistic conditions.
